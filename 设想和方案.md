## 一、 项目背景与当前痛点 (Background & Pain Points)

在日常的跨网络协作、私有化部署、远程排查以及边缘设备（IoT）管理中，**内网穿透（NAT Traversal）与网络代理**是不可或缺的底层基础设施。目前市面上的主流方案存在以下痛点，难以完全契合我们日益增长的业务和运维需求：

1. **老牌标杆 `ehang-io/nps` 已陷入停滞**：该项目曾因“开箱即用 + 自带精美 Web 面板”极受欢迎，但目前已停更数年，积累了大量未修复的 Bug，且底层网络协议无法适应现代复杂的网络环境。
2. **主流竞品（如 `frp`）体验断层，运维心智负担重**：以 `frp` 为代表的优秀工具虽然底层性能强悍，但本质上是极客向的纯命令行工具，高度依赖繁琐的配置文件（`.ini` / `.toml`）。缺乏高度集成的现代化 Web 可视化管理能力，非专业人员使用门槛极高。
3. **“转发”与“管控”割裂（运维黑盒）**：现有的代理工具仅仅是无情的“流量搬运工”。服务端对远端内网机器的**真实物理健康状态（CPU、内存、网络IO等）**一无所知，缺乏全局视角的监控能力。

**结论**：我们需要重新设计并自研一款**兼具工业级转发性能、极简部署体验与全局状态监控**的新一代网络管控枢纽。

## 二、 产品愿景与核心定位 (Product Vision)

本项目旨在打造一个**“轻量级管控中心 (C2) + 高性能网络隧道”**双效合一的统一平台。

1. **零门槛的现代化管控大盘**：彻底告别配置文件。所有的隧道分发、权限控制、流量审计均通过极具现代感、数据可视化的 Web 面板完成。
2. **“代理 + 探针”双剑合璧**：客户端（Agent）不仅负责打通内外网流量，更是一个轻量级的系统探针，实时向控制中心上报宿主机运行状态。
3. **极致的开箱即用体验**：无论服务端还是客户端，对外均交付**“单文件、零外部依赖”**的可执行程序，将部署实施成本降至最低。

## 三、 架构设计与技术选型：行业黄金标准 (Architecture Design)

经过严密的技术预研，本项目决定摒弃传统的跨语言混编方案（如 Node.js + Go），采用目前高性能网络基建领域的黄金标准架构：**全栈 Golang (服务端+客户端) + 独立现代化前端 (React/Vue)**。

### 1. 核心技术栈拆解

* **服务端 (Server)**：`Golang` (提供核心网络调度与 API) + `纯 Go 版 SQLite` (轻量级本地存储，免除用户配置 MySQL 的繁琐)。
* **客户端 / Agent (Client)**：`Golang` (极致的跨平台编译能力) + `gopsutil` (跨平台机器硬件状态采集库)。
* **Web 管理面板 (Frontend)**：`Vite` + `React/Vue` + `TailwindCSS` + `Echarts` (用于渲染实时机器监控波浪图)。

### 2. 核心网络拓扑 (控制面与数据面分离)

* **控制通道 (Control Channel)**：Agent 启动后与 Server 建立持久的长连接（WSS 或 TCP）。该通道**仅用于**传输轻量级信令（心跳保活、终端状态上报、下发开通隧道的指令）。
* **数据通道 (Data Channel)**：当产生真实外部访问时，基于控制通道的指令唤醒，Agent 与 Server 建立高速数据通道，进行底层 TCP/UDP 报文的无脑大流量搬运。

## 四、 本架构的压倒性优势 (Why this Stack?)

本选型能够为团队研发效能和最终产品体验带来极大的红利，也是本次重构的核心底气：

### 🏆 优势一：彻底攻克“跨语言多路复用”难题，性能跃升

内网穿透的核心难点在于“如何在一个 TCP 长连接里，并发传输成千上万个独立访客的网络请求”。
由于双端均采用纯 Go 实现，我们**无需手搓复杂的底层二进制帧协议**，可直接无缝引入成熟的工业级多路复用库（如 `HashiCorp/yamux` 或 `smux`）。寥寥数行代码，即可在单条物理连接上虚拟出无数个互相隔离的 Stream，彻底消除了跨语言通信的内存泄漏与拆包粘包风险，大幅提升大流量并发吞吐率。

### 🏆 优势二：极致的代码复用，杜绝通讯 Bug (Monorepo 优势)

采用单体仓库（Monorepo）模式，服务端与客户端**直接物理共享同一个核心协议包 (`pkg/protocol`)**。
所有的通讯指令、加密算法、特别是**复杂的机器状态上报结构体（Struct）**只需编写一次。改一处代码，双端同时生效，从根本上杜绝了联调时 JSON 字段对不上导致的低级 Bug。

### 🏆 优势三：“魔法”打包，杀手级的单文件交付 (`//go:embed`)

前端采用 React/Vue 开发势必带来部署上的复杂性（需额外配置 Nginx），但这对于轻量级工具是致命伤。
我们将利用 Go 1.16+ 提供的 **`//go:embed`** 神级特性。在编译阶段，**直接将前端构筑出的 `dist` 静态网页目录物理嵌入到 Server 端的二进制可执行文件中**。

* **最终体验**：用户/实施人员拿到手只有一个几十 MB 的 `server` 程序。双击运行后，底层的穿透服务启动，同时浏览器访问 `http://ip:端口`，直接就能看到精美的 Web 后台面板。**环境配置门槛降至绝对零度。**

## 五、 项目工程结构规划 (Monorepo Layout)

系统将由三个逻辑项目组成，但在同一个 Git 仓库下统一协同研发：

```text
Next-NPS/
├── cmd/
│   ├── server/       # 项目1：服务端主入口 (内置嵌前端静态资源的逻辑)
│   └── client/       # 项目2：客户端/探针 主入口
├── pkg/
│   ├── protocol/     # 💎 【核心价值】双端共享通讯协议、指令集、探针状态结构体
│   └── mux/          # 基于 yamux/smux 的底层多路复用封装
├── internal/
│   ├── server/       # 服务端核心逻辑 (API 路由、鉴权、隧道调度、SQLite CRUD)
│   └── client/       # 客户端核心逻辑 (探针硬件数据采集、本地网络端口打洞)
├── web/              # 项目3：纯前端项目目录
│   ├── src/          # React/Vue 源码
│   ├── package.json
│   └── dist/         # 构建产物 (供 cmd/server 编译时强行嵌入)
├── go.mod            # 双端公用依赖管理
└── Makefile          # 一键构建流水线 (Build Web -> Embed to Server -> Build Client)

```

## 六、 演进路线图 (Phase Roadmap)

为控制风险，快速验证核心价值，建议分三个阶段进行敏捷迭代：

* **Phase 1：MVP 核心跑通（骨架与信令）**
* 制定统一的 控制通道 (Control Channel) 协议规范。
* 跑通 Agent 连接 Server 的控制流，实现基础系统状态（CPU/内存）的采集与定时上报。
* 实现单个 TCP/HTTP 端口的无脑转发映射。


* **Phase 2：核心竞争力注入（性能与体验）**
* 引入 `yamux` 将底层数据通道升级为工业级多路复用长连接。
* 跑通 React 前端构建，并使用 `go:embed` 成功嵌入 Server 端。
* 接入纯 Go 版 `SQLite`，实现配置、规则、多用户数据的本地持久化。


* **Phase 3：数据可视化与商业/企业级特性**
* 完善 Web 面板，接入 Echarts 绘制设备在线状态与 CPU/网络波浪监控大屏。
* 实现泛域名 HTTP/HTTPS 智能路由自动分配。
* 更细粒度的流量限速、数据统计与告警推送。

### 💡 附：给您的研讨会汇报小提示 (Tips)

1. **强调“多路复用”**：如果有人问“为什么要用 Go 换掉 Node.js？”，请重点抛出“多路复用”这个概念。说明跨语言处理底层二进制拆包、粘包极易出现内存泄漏，而“双端纯 Go”不仅有现成的工业级方案（Yamux），且 `io.Copy` 的零拷贝转发性能无敌。
2. **放大 `go:embed` 的震撼感**：很多非 Go 语言栈的同事/领导对“把前端整个网页塞进后端二进制文件”这个特性会感到非常惊艳。**一定要重点讲**，这是本方案在部署体验上碾压传统前后端分离方案的最大卖点，极大降低了实施成本。
3. **展示可视化大屏的价值**：在会上可以着重描绘一下前端 Echarts 展现“全网节点 CPU 折线图”的画面感。这个“探针”的差异化特性最容易打动运维主管和产品经理，因为它把一个底层的黑盒工具变成了直观的 IT 资产管理平台。
