## 2. 评审结论摘要

### 2.1 一句话结论

**“数据通道 WebSocket 化 + 全链路 TLS 化”方向正确，值得推进；但原方案中关于 TOFU、证书生命周期、反向代理信任边界和迁移策略的表述不够严谨，不能原样直接落地。**

### 2.2 修订后的总体判断

- 推荐继续采用“**控制面与数据面统一到标准 HTTP/WS 生态**”作为主线方向。
- 推荐采用“**TLS 三态能力**”，但必须修正为**安全边界明确、可运维、可迁移**的版本。
- 推荐将实施拆成多个阶段，而不是一步到位替换底层传输栈。

### 2.3 评审裁定

**有条件通过。**

通过条件：

1. ✅ 修正 TOFU 的安全表述，不能将其描述为"首次连接即可绝对防 MITM"。
2. ✅ `auto` 模式下证书必须可持久化，不能仅存于内存。（已实现：`loadOrGenerateAutoTLS` 持久化到 `~/.netsgo/tls/`）
3. ✅ `off` 模式必须补充受信任反代模型，不能盲信 `X-Forwarded-Proto`。（已实现：`TLSConfig.TrustedProxies` CIDR 列表）
4. 补齐 `/ws/data` 与 `yamux` 适配层设计。（阶段 2 范围）
5. ✅ 补齐双栈迁移策略、回滚策略和验收标准。（已写入 `docs/implementation-plan.md`）

## 4. 问题根因再识别

### 4.1 P1 / P2 的根因不是“没开 TLS”这么简单

P1 与 P2 的真正根因是：

1. **控制面和数据面的传输安全能力不统一**。
2. **数据面使用自定义裸 TCP 入口，天然不兼容标准 HTTP 反向代理生态**。
3. **一旦用户想通过 Nginx/Caddy 做 HTTPS 收口，控制面和数据面会被拆成两套入口能力，运维复杂且容易出错**。

### 4.2 当前架构的结构性问题

当前单端口 + Peek 分流的架构虽然功能上可用，但会带来以下长期问题：

- 接入层难以标准化治理。
- 七层代理无法接住裸 TCP 数据面。
- TLS 无法自然覆盖全部流量。
- 日志、审计、限流、WAF、CDN、LB 等能力无法统一复用。
- 自定义分流协议会持续增加维护和测试成本。

因此，P1 / P2 的修复不应只理解为“在当前监听器外面包一层 TLS”，而应理解为：

> **把 NetsGo 的外部接入形态收敛到标准 HTTPS / WSS 生态中。**

## 5. 方案对比与取舍

### 5.1 方案 A：保持现有数据面，外层补一层 TLS

做法：

- 继续保留 `PeekListener + 魔数 0x4E + 裸 TCP + yamux`
- 在外层使用 `tls.Listener`

优点：

- 改动面相对小
- 对现有 Agent 与数据面逻辑侵入较低

缺点：

- 依然不能自然接入七层反代生态
- 对用户部署和排障不友好
- 长期仍保留两类协议入口
- 不能从根上消除接入复杂性

结论：**可作为临时止血方案，但不建议作为长期主架构。**

### 5.2 方案 B：数据面改造成 `/ws/data`，统一到 WebSocket

做法：

- 废弃外部暴露的魔数裸 TCP 数据入口
- 数据通道也走标准 WebSocket 升级
- 在 WebSocket 二进制帧上承载 `yamux`

优点：

- 与标准 HTTPS / WSS / 反代生态天然兼容
- 路由和接入层能力统一
- 运维、观测和安全策略更一致

缺点：

- 需要实现稳定的 WS ↔ 字节流适配层
- 小包和高频场景会有额外帧开销与 CPU 开销
- 迁移期需要双栈兼容

结论：**推荐作为目标长期架构。**

### 5.3 方案 C：应用层自定义加密/MAC/防重放

做法：

- 继续使用明文 TCP / WS 作为底层通道
- 在消息层自行实现加密、MAC、nonce、序号、时间窗

优点：

- 理论上可以做到“无 TLS 的协议自保护”

缺点：

- 开发复杂度高
- 密码学设计与实现风险高
- 与标准生态脱节
- 测试与审计成本显著升高

结论：**不建议作为当前阶段主方案。**

### 5.4 推荐策略

推荐采用：

> **以方案 B 为目标态，以阶段性 TLS 能力建设作为过渡手段。**

也就是：

1. 先补齐 TLS 基础能力，快速降低明文风险。
2. 再把数据面迁移为 `/ws/data`。
3. 最后清退旧的 Peek + 魔数协议。

## 6. 修订后的推荐方案

### 6.0 地址模型定义（新增，必须统一）

P1 修复不能只讨论 TLS 和 `/ws/data`，还必须统一“服务地址”在系统中的语义。否则即使底层实现正确，前端、CLI、生成命令和部署说明仍然会彼此打架。

#### 当前现状

当前系统内同时存在两种接近但不完全一致的地址语义：

1. **客户端 `--server` 参数**：当前代码实际期望的是 `ws://host:port` 形式的 WebSocket 基地址。
2. **管理端 `server_addr` 配置**：更接近“用户对外公开访问的完整地址”，通常由初始化页面以 `http://` 或 `https://` 形式填写。

这两个语义在明文时代还能勉强共存，但在引入 `https://` / `wss://` 之后，如果不统一，后果会非常直接：

- 前端可能生成错误的 Agent 启动命令；
- 客户端可能把 `https://...` 当成 WebSocket 地址直接拼接；
- 文档示例、CLI、后台展示和真实连接行为相互不一致。

#### 修订后的统一定义

推荐将系统中的“服务地址”统一定义为：

> **`server_addr` 表示公共 HTTP 基地址，而不是 WebSocket 专用地址。**

例如：

- `http://127.0.0.1:8080`
- `https://tunnel.example.com`

在该定义下，客户端或前端生成连接地址时按如下规则推导：

1. 若 `server_addr` 以 `http://` 开头，则控制通道映射为 `ws://.../ws/control`，数据通道映射为 `ws://.../ws/data`。
2. 若 `server_addr` 以 `https://` 开头，则控制通道映射为 `wss://.../ws/control`，数据通道映射为 `wss://.../ws/data`。
3. Web 面板和 REST API 始终沿用 `server_addr` 本身的协议语义。

#### 实施约束

P1 实施过程中，所有新增或修改的代码、测试、CLI 帮助文本、前端文案都必须遵守这个统一定义。

### 6.1 目标架构

目标形态如下：

- Web 面板：`https://host/`
- API：`https://host/api/...`
- 控制通道：`wss://host/ws/control`
- 数据通道：`wss://host/ws/data`

所有外部入口统一为：

- 标准 HTTP/1.1 或 HTTP/2 体系内的 HTTPS / WSS
- 单端口可接入标准反向代理
- 传输层安全交由 TLS 提供

### 6.2 对 P1 / P2 / P4 / P11 的影响边界

采用该方案后：

- **P1（控制通道明文）**：可直接解决。
- **P2（数据通道明文）**：可直接解决。
- **P4（消息级防篡改与防重放）**：在链路层大幅缓解，但不等于所有业务语义层 replay 都天然消失。
- **P11（敏感材料返回前端）**：仅缓解网络窃听，不解决浏览器端暴露问题。

必须明确：

> **TLS 是传输安全底座，不是浏览器会话安全和前端敏感信息治理的替代品。**

### 6.3 P1 修复范围定义（新增，防止理解跑偏）

P1 的修复目标不应被狭义理解为“让服务端支持 TLS 启动”。

P1 的**准确修复范围**应定义为：

1. 控制通道从默认明文 `ws://` 进入可控的 `wss://` 模型；
2. 服务端具备明确的 TLS 部署模式（`custom` / `auto` / `off`）；
3. 地址模型统一，`server_addr`、前端展示、CLI 参数、生成命令使用同一套语义；
4. 阶段性过渡时，不允许把临时架构误当成长期终态；
5. 为数据面后续收敛到 `/ws/data` 预留稳定路径；
6. 测试、示例、部署说明与真实实现保持一致。

也就是说：

> **P1 的修复不是单一代码点修复，而是一次“控制面传输安全 + 接入模型 + 迁移路径”三者联动的架构修复。**

## 7. TLS 三态策略（修订版）

原方案中的“三态 TLS”思路可以保留，但需要重新定义边界。

### 7.1 `tls.mode = "custom"`

含义：

- 用户显式提供 `cert_file` 与 `key_file`
- Server 直接启动 HTTPS / WSS

适用场景：

- 正式生产环境
- 有域名、有 ACME/Let’s Encrypt 或企业证书的用户

评价：

- 这是**最标准、最推荐**的生产模式
- 客户端使用标准 PKI 校验，无需 TOFU

### 7.2 `tls.mode = "auto"`

含义：

- Server 自动生成自签名证书
- **证书或签发根必须持久化到磁盘**
- Agent 首次连接时可采用 TOFU 记录证书指纹或公钥指纹

适用场景：

- 纯 IP 用户
- 无域名、无外部证书条件的轻量部署
- 内网实验室、家庭场景

关键修正：

1. **不能只在内存生成证书。** 否则服务重启后指纹漂移，TOFU 无法稳定工作。
2. TOFU **不能防首次连接 MITM**，只能降低后续连接被冒充风险。
3. 自动生成证书时，必须支持：
    - IP SAN
    - Hostname SAN
    - 合理有效期
    - 可控轮换策略
4. 客户端必须定义 pin mismatch 的处理策略：
    - 默认拒绝继续连接
    - 允许用户显式确认更新信任
    - 记录审计日志

评价：

- 这是“纯 IP 场景下比明文强得多”的实用模式
- 但它不是浏览器绿锁，也不是严格意义上的首次连接零信任安全

### 7.3 `tls.mode = "off"`

含义：

- NetsGo 自身不终止 TLS
- 由前置反向代理负责 HTTPS / WSS

适用场景：

- Nginx / Caddy / Traefik / 云负载均衡前置

关键修正：

1. 不能简单写成“信任 `X-Forwarded-Proto: https`”。
2. 必须引入**受信任代理列表**，例如：
    - `trusted_proxies = [127.0.0.1/32, 10.0.0.0/8]`
3. 只有当请求来源于 trusted proxy 时，才解析 `X-Forwarded-*` 头。
4. 最佳实践是：
    - NetsGo 只监听本地回环或内网地址
    - 由反代对公网暴露

评价：

- 对专业用户是最佳部署方式之一
- 但前提是**代理信任边界被严格配置**

## 8. Agent 信任模型（修订版）

### 8.1 `custom` / 正式证书场景

- 走标准证书链校验
- 客户端按域名/IP 校验证书
- 不依赖 TOFU

### 8.2 `auto` / 自签名场景

- 首次连接：Agent 记录指纹（TOFU）
- 后续连接：Agent 校验指纹是否一致
- 指纹变化：默认视为高风险，拒绝连接并告警

### 8.3 TOFU 的边界

TOFU 能做到：

- 避免“后续所有连接都无条件相信网络对端”
- 在运维简单场景下大幅优于明文

TOFU 做不到：

- 解决首次连接信任建立问题
- 替代标准 PKI
- 让浏览器对自签证书消除安全告警

因此，文档与产品说明中应使用如下表述：

> `auto` 模式为“低配置高收益”的传输加密方案，而不是“与正规证书等价”的生产级信任体系。

## 9. `/ws/data` 适配层设计要求

原方案中“编写几十行 Adapter”这一说法过于乐观。这里明确该适配层应作为独立设计项。

### 9.1 目标

将 WebSocket 二进制消息封装为一个对 `yamux` 透明的 `io.ReadWriteCloser`。

### 9.2 需要满足的能力

1. **字节流语义模拟**
    - WebSocket 是消息边界协议
    - `yamux` 需要稳定的连续字节流

2. **并发约束处理**
    - gorilla/websocket 对并发读写有明确约束
    - 适配层必须集中管理单写协程或写锁

3. **关闭语义**
    - 正常关闭
    - 异常关闭
    - 读 EOF / 写失败
    - Close frame 与底层流结束的一致性

4. **超时与 keepalive**
    - 代理/LB/云厂商可能对空闲 WS 有超时限制
    - 需要 ping/pong 与读写 deadline 策略

5. **性能控制**
    - 大包拆分
    - 小包聚合策略
    - 内存池/缓冲区复用
    - backpressure 控制

### 9.3 结论

`/ws/data` 适配层可以做，但必须被视为一项**中等复杂度协议适配工作**，而不是零成本封装。

## 10. 迁移策略（必须项）

为了避免旧 Agent 全量失联，推荐采用双栈迁移。

### 10.1 阶段性兼容策略

Server 在过渡期同时支持：

1. **旧数据面**：`PeekListener + 0x4E + TCP + yamux`
2. **新数据面**：`/ws/data + WSS + yamux`

Agent 能力协商建议：

- 新版本 Agent 优先尝试 `/ws/data`
- 失败时可按策略回退到旧协议（仅在兼容窗口期内）

### 10.2 清退条件

只有在满足以下条件后，才应移除旧协议：

- 新 Agent 占比达到预设阈值
- `/ws/data` 在核心场景稳定运行一段观察期
- 有明确回滚预案
- 文档和部署指引已全部更新

### 10.3 回滚策略

若新数据面出现重大兼容或性能问题，应支持：

- 服务端配置开关回退到旧协议优先
- Agent 配置强制使用旧协议
- 保留协议级兼容窗口直到问题修复

### 10.4 阶段 1 的过渡边界（新增，必须明确）

阶段 1 的目标是“TLS 止血”，不是“最终架构已经完成”。

必须明确以下边界：

1. **阶段 1 允许存在过渡型实现**，包括在不完全改变现有数据面的前提下，先让控制通道进入 TLS / WSS。
2. **阶段 1 不得被定义为长期终态**。只要数据面仍停留在旧的 `PeekListener + 0x4E + yamux` 模型上，P1 的长期架构目标就还没有完全达成。
3. **阶段 2 的 `/ws/data` 收敛是正式范围，不是可无限延期的“优化项”**。
4. 评审、排期和对外说明中必须统一使用以下表述：
    - 阶段 1：止血与基础能力建设；
    - 阶段 2：架构收敛；
    - 阶段 3：迁移与清退。

交付物：

- Server 支持 `custom` / `auto` / `off` 三态配置
- `auto` 模式下证书持久化
- Client 支持 `wss://`
- 基础证书校验与 pinning 能力

验收点：

- 控制通道可在 TLS 下稳定连接
- 明文控制通道在默认配置下不再是唯一选项
- `auto` 模式重启后指纹稳定

### 阶段 2：`/ws/data` 新数据面实现（收敛阶段）

目标：完成数据面 WebSocket 化。

交付物：

- Server 提供真实 `/ws/data`
- Agent 能通过 `/ws/data` 建立数据通道
- WS ↔ 字节流适配层
- 新数据面集成测试

验收点：

- TCP 代理可稳定运行
- 长连接和大流量场景无明显功能回退
- 支持在 Nginx / Caddy 后正常工作

### 阶段 3：双栈兼容与迁移

目标：平滑替换旧协议。

交付物：

- 新旧数据面并存
- Agent 具备能力协商/回退策略
- 迁移观察指标

验收点：

- 升级过程中旧 Agent 不会被立即全量踢下线
- 线上可观测新旧协议使用比例

### 阶段 4：安全收口与后续增强

目标：补齐 TLS 之外的残余高风险项。

交付物：

- 管理端 token 存储策略升级（优先向 Cookie/服务端 session 收敛）
- 敏感材料回传与展示窗口收缩
- 更细粒度的审计与异常告警

验收点：

- P5 / P11 有独立整改闭环
- 文档不再暗示“TLS 已经顺带解决所有高风险项”

## 12. 风险矩阵

| 风险项 | 描述 | 概率 | 影响 | 风险等级 | 缓解策略 |
| --- | --- | --- | --- | --- | --- |
| 首次连接 MITM | `auto` + TOFU 场景下首次连接仍可能被劫持 | 中 | 高 | 高 | 文档明确边界；提供指纹确认；生产优先 `custom` |
| 证书指纹漂移 | 自签证书若不持久化，重启后会频繁变更 | 高 | 高 | 高 | 证书或根证书持久化；明确轮换流程 |
| 反代头伪造 | `off` 模式若盲信 `X-Forwarded-*` 头会被绕过 | 中 | 高 | 高 | 引入 `trusted_proxies`；限制监听地址 |

| 运维复杂度上升 | 新增证书、信任、代理配置带来使用门槛 | 中 | 中 | 中 | 提供默认模式、清晰部署文档与诊断信息 |

## 13. 验收标准（Definition of Done）

### 13.0 P1 精确验收范围（新增）

P1 的完成判定必须包含**代码、配置、前端、测试、文档**五个层面，而不是只看服务端是否能跑 TLS。

必须同时覆盖：

1. **代码层**：Server、Client、地址推导、TLS 模式、必要的过渡兼容逻辑。
2. **配置层**：`server_addr`、`tls.mode`、证书参数、trusted proxies 等语义清晰且可用。
3. **前端层**：初始化页面、接入说明、快速接入命令与目标架构一致。
4. **测试层**：至少有 TLS / WSS / 地址推导 / 迁移兼容的覆盖。
5. **文档层**：README、问题文档、方案文档、实施计划同步更新。

### 13.1 功能验收

1. Server 可在 `custom` / `auto` / `off` 三种模式下启动。
2. Agent 可通过 `wss://` 建立控制连接。
3. 新版 Agent 可通过 `/ws/data` 建立数据通道。
4. 反代（Nginx/Caddy）后，控制面与数据面均可工作。
5. 兼容窗口期内，旧 Agent 仍可按策略接入。
6. 前端生成的 Agent 接入命令与实际客户端参数语义一致。
7. `server_addr` 到 `ws/wss` 的映射规则在客户端和前端中一致实现。

### 13.2 安全验收

1. 默认推荐配置下，不再只有明文控制通道。
2. `auto` 模式重启后证书指纹保持稳定。
3. `auto` 模式下发生 pin mismatch 时，客户端默认拒绝静默继续。
4. `off` 模式仅信任来自 trusted proxy 的转发头。
5. 文档明确 TOFU 边界，不再出现“首次连接绝对防 MITM”类表述。

### 13.3 兼容性验收

1. 新旧协议切换对现网 Agent 的影响可控。
2. 新旧协议都有明确可观测指标和日志。
3. 出现问题时，可通过配置回退。
4. 阶段 1 的过渡方案不会在文档和实现层被误标为长期终态。

### 13.4 性能验收

以下指标应以项目现有 benchmark 与真实测试环境测量为准，不得拍脑袋承诺：

1. 常见 TCP 穿透场景功能稳定。
2. 长连接场景无异常断连或明显吞吐塌陷。
3. 小包/高频/UDP 场景有独立测试数据支持是否可接受。

## 14. 明确不由本方案单独解决的问题

为了避免误解，这里单独列出本方案不直接闭环的问题。

### 14.1 P5：管理端 token 存在 localStorage

TLS 无法解决浏览器内的 XSS / 本机被控 / 恶意扩展读取问题。

### 14.2 P11：敏感材料返回前端

TLS 只能防止网络窃听，不能消除前端暴露窗口。

### 14.3 更深层业务防重放

如果未来存在离线任务重投、异步命令总线或跨连接复用控制命令，则仍可能需要业务级 nonce / sequence / idempotency 设计。

---

## 15. 最终建议

最终建议如下：

1. **继续推进本方案，但按修订版执行。**
2. **先完成 TLS 止血，再推进 `/ws/data` 收敛。**
3. **双栈迁移必须作为正式范围，不得省略。**
4. **P5 / P11 必须独立进入后续安全整改计划。**

简而言之：

> 这不是“是否要做”的问题，而是“必须用正确的方式做”。

在修订后的边界下，该方案可以作为 NetsGo 从“功能可用”迈向“可在公网较稳妥部署”的关键基础设施改造来推进。
