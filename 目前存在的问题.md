# 目前存在的问题

## 1. 文档目的

本文用于说明当前版本在“服务端 - 客户端交互安全”方面已经确认存在的问题，供项目维护者、贡献者和后续接手者统一理解现状。

这不是修复方案文档，而是问题定义文档。目标是回答三件事：

1. 当前到底有哪些问题。
2. 这些问题为什么是问题。
3. 哪些问题必须优先解决，才能让项目在“无 TLS 也能使用”的目标下尽量安全。

## 2. 背景与产品目标

- 项目定位是开源内网穿透工具。
- 产品必须允许用户在没有 TLS 的情况下也能使用。
- 因此，安全设计不能完全依赖 TLS；协议本身必须承担一部分安全职责。
- 这并不意味着“没有 TLS 就允许明文裸奔”，而是意味着：即使没有 TLS，协议层也应该尽可能提供认证、保密、完整性和防重放能力。

换句话说：

- 有 TLS 时，TLS 应该是增强项。
- 没有 TLS 时，协议层也必须有最基本的自保护能力。

## 3. 当前总体结论

### 3.1 一句话结论

当前版本更接近“功能可用”，但还不能满足“无 TLS 条件下的安全传输”目标。

### 3.2 问题本质

当前最核心的问题不是“没有 TLS”本身，而是：

- 控制通道没有建立消息级安全保护。
- 数据通道没有和认证态强绑定。
- 管理端会话在浏览器侧的存储与使用方式偏弱。
- 多个问题可以串联形成完整攻击链。

### 3.3 风险等级

- 总体风险等级：高。
- 如果按当前实现直接暴露到公网，不建议视为“安全可上线”状态。
- 如果只在受控内网或测试环境中使用，风险会下降，但协议层缺口依然存在。

## 4. 威胁模型

为了让后续讨论有统一前提，这里明确当前需要考虑的攻击者能力。

### 4.1 需要考虑的攻击者能力

1. 能监听或篡改客户端与服务端之间的网络流量。
2. 能发起伪造连接，尝试冒充 Agent 或插入数据通道。
3. 能对管理端登录接口、Agent 认证接口做暴力尝试。
4. 能在浏览器端通过 XSS 或本地环境拿到管理员 token。
5. 能利用已泄露的 token、AgentID、Key 继续横向利用。

### 4.2 当前最小安全目标

即使用户不启用 TLS，也至少应该做到：

1. 控制通道身份可验证。
2. 控制消息不可被静默篡改。
3. 控制消息不可被无限重放。
4. 数据通道必须绑定当前认证态，而不是只靠公开标识。
5. 长期凭据不能在网络上直接裸传后长期复用。
6. 管理员会话被盗后，攻击成本不能过低。

## 6. 逐项问题详解

### P1. 控制通道默认是明文 WebSocket，认证材料和控制消息可被窃听或篡改

#### 问题描述

客户端默认使用 `ws://` 连接服务端，服务端也以普通 `http.Server` 提供控制面能力，没有内建 TLS 保护。

#### 为什么这是问题

控制通道承载了以下敏感内容：

- 管理端登录后的 token。
- Agent 连接时使用的 Key/Token。
- 心跳。
- 探针数据。
- 建隧道、暂停、恢复、关闭等控制指令。

如果这条链路是明文的，那么中间人不仅能看见内容，还可能直接篡改内容。

#### 可能的攻击方式

- 窃取管理员或 Agent 的认证材料。
- 篡改控制消息，例如伪造建隧道、关隧道、状态同步等。
- 做连接劫持或流量回放。

#### 影响

这是无 TLS 场景下最基础的保密性与完整性缺口。

### P2. 数据通道是裸 TCP，当前没有独立的加密、完整性校验和防重放能力

#### 问题描述

数据通道不是 TLS，也没有协议层的加密、MAC、nonce、序号、时间窗等机制。

#### 为什么这是问题

在“允许无 TLS 使用”的产品目标下，数据通道如果只是裸 TCP，那么：

- 数据是否被窃听，无法控制。
- 数据是否被篡改，无法判断。
- 老流量是否被重放，无法判断。

#### 可能的攻击方式

- 中间人直接观察和修改数据流。
- 回放旧握手或旧消息，诱导协议进入错误状态。
- 构造伪造帧干扰传输。

#### 影响

如果不解决这个问题，就无法说系统具备“无 TLS 也能尽量安全传输”的能力。

### P4. 控制面协议没有消息级签名、序号、随机数或时间窗，缺少防篡改与防重放机制

#### 问题描述

当前控制消息本质上是 JSON 封装，没有看到：

- 消息签名
- MAC
- nonce
- 时间戳校验
- 序列号或递增计数器

#### 为什么这是问题

在没有 TLS 的条件下，协议层必须自己解决“这条消息有没有被改过”和“这是不是旧消息重放”的问题。

当前没有这些机制，就意味着：

- 消息来源可信度不足。
- 消息完整性无法证明。
- 消息新鲜度无法证明。

#### 可能的攻击方式

- 篡改控制消息字段。
- 重放旧的认证或控制消息。
- 伪造看起来格式正确的消息。

#### 影响

这是“无 TLS 安全传输”能否成立的基础问题之一。

### P5. 管理端 JWT 保存在浏览器 localStorage，XSS 一旦出现会直接升级成会话接管

#### 问题描述

前端把管理员 token 持久化到 `localStorage`，之后每次请求手动拼到 `Authorization` 头里。

#### 为什么这是问题

这种方式的典型风险是：

- 只要页面未来出现 XSS，攻击脚本就可以直接读取 token。
- 一旦 token 泄露，而后端又没有真正的设备绑定，那么会话可以在别处被直接复用。

#### 可能的攻击方式

- 通过 XSS 读出 `localStorage` 中的 token。
- 通过恶意浏览器环境、扩展、共享机器等场景窃取 token。

#### 影响

管理后台一旦失守，攻击者就能进一步查看 Agent 列表、操作隧道、读取系统信息，甚至为后续数据面攻击创造条件。

### ~~P13. SSE 事件流端点设置了 `Access-Control-Allow-Origin: *`~~ ✅ 已修复

#### 问题描述

SSE 端点 `/api/events` 的响应中设置了 `Access-Control-Allow-Origin: *`，完全放开了跨域访问限制。

#### 为什么这是问题

虽然 SSE 端点受 JWT 鉴权保护，但 CORS 响应头为 `*` 意味着：一旦攻击者通过其他手段获取了 token（如 P5 的 localStorage 泄露），就可以从任意第三方域发起 SSE 连接，实时读取 Agent 上下线、隧道变更、探针数据等敏感事件流。

这与 P9（WebSocket Origin 放开）属于同类问题，但发生在不同的端点上。

#### 可能的攻击方式

- 第三方恶意页面通过窃取的 token 建立 SSE 连接，持续监听系统事件。
- 获取 Agent 列表、隧道配置变更等运维敏感数据。

#### 影响

放大 token 泄露后的信息暴露面。

#### 代码证据

- `internal/server/events.go:96`

### ~~P14. HTTP Server 未设置超时参数，易受 Slowloris 等慢速攻击~~ ✅ 已修复

#### 问题描述

`http.Server` 创建时没有配置 `ReadTimeout`、`WriteTimeout`、`ReadHeaderTimeout` 和 `IdleTimeout`，所有超时均为 Go 默认的无限等待。

#### 为什么这是问题

对公网暴露时，攻击者可以利用 Slowloris 等慢速攻击手段：持续发送不完整的 HTTP 请求头或极慢地传输请求体，大量消耗服务端连接和 goroutine 资源，最终导致拒绝服务。

#### 可能的攻击方式

- Slowloris 攻击：建立大量连接，每个连接每几秒发送一个字节的请求头，永远不完成请求。
- Slow POST 攻击：发送一个声称有很大 Content-Length 的请求，然后极慢地传输 body。

#### 影响

在连接数有限的环境下，少量攻击源即可耗尽服务端的可用连接池，导致合法用户和 Agent 无法访问。

#### 代码证据

- `internal/server/server.go:253`

### ~~P15. 无优雅关闭（Graceful Shutdown）机制~~ ✅ 已修复

#### 问题描述

进程在收到 SIGTERM 或 SIGINT 信号时直接退出，没有实现优雅关闭逻辑。

#### 为什么这是问题

进程被终止时不会：

- 等待正在处理的 HTTP 请求完成。
- 正常关闭与 Agent 的 WebSocket 和数据通道连接。
- 确保持久化数据完整写入。

#### 可能的影响

- 持久化数据（admin.json、tunnels.json）可能在写入中途被截断。
- Agent 侧无法收到正常的关闭通知，会误判为网络中断并进入不必要的重连循环。

#### 影响

这更多是健壮性和运维体验问题，而非直接的安全漏洞，但在生产环境中值得修复。

#### 代码证据

- `cmd/netsgo/cmd_server.go`（缺少 signal 处理逻辑）

### ~~P16. 控制通道 WebSocket 认证阶段无超时~~ ✅ 已修复

#### 问题描述

Agent 通过 WebSocket 连接后，服务端在 `handleAuth` 中等待认证消息时没有设置读超时。

#### 为什么这是问题

恶意客户端可以建立大量 WebSocket 连接后永远不发送认证消息。每个连接都会占用一个 goroutine 和一个 WebSocket 对象，长期不释放。

#### 可能的攻击方式

- 批量建立 WebSocket 连接到 `/ws/control`，建立后不发送任何数据。
- 服务端每个连接都阻塞在 `conn.ReadJSON(&msg)` 上，消耗资源。

#### 影响

可能导致服务端 goroutine 泄漏和资源耗尽，影响合法 Agent 的连接能力。

#### 代码证据

- `internal/server/server.go:450`
- `internal/server/server.go:451`

## 7. 典型攻击链

下面列出几个现实中更容易发生的组合风险，帮助理解为什么这些问题不能孤立看待。

### 攻击链 A：明文控制面 -> 管理端 token 泄露 -> 后台被接管

步骤：

1. 攻击者监听管理端与服务端之间的明文流量。
2. 拿到管理员登录 token。
3. 直接调用后台 API。
4. 获取 Agent 列表、查看主机信息、操作隧道、创建新 Key。

形成条件：P1 + P5 + P6。

### 攻击链 B：管理端 token 泄露 -> 获取 AgentID -> 伪造或抢占数据通道

步骤：

1. 攻击者通过 XSS、本地环境或明文流量拿到管理员 token。
2. 调用 `/api/agents` 获取在线 Agent 信息和 `agent_id`。
3. 利用数据通道仅依赖 `AgentID` 握手的弱点，尝试构造伪造数据面连接。

形成条件：P3 + P5 + P6。

### 攻击链 C：未初始化实例直接被外部抢注

步骤：

1. 实例首次部署后暴露在网络上。
2. 管理员尚未完成初始化。
3. 外部访问者先调用初始化接口。
4. 攻击者成为首个管理员。

形成条件：P8。

### ~~攻击链 D：未初始化实例 + 固定 JWT 密钥 -> 伪造管理员身份~~ ✅ P12 已修复，此攻击链已阻断

步骤：

1. 实例首次部署后暴露在网络上，管理员尚未初始化。
2. 攻击者知道 JWT Secret 是硬编码的 `"netsgo-dev-fallback-secret"`。
3. 攻击者用该密钥签发伪造的 JWT，尝试调用受保护的 API。
4. 虽然大部分 API 在未初始化时会因 session 不存在而失败，但如果未来有新的 API 分支不检查 session，就可能被绕过。

形成条件：P8 + ~~P12~~。

注：P12 已修复——硬编码 fallback 密钥已移除，未初始化时 `GetJWTSecret()` 会直接返回错误而非使用固定密钥。此攻击链已阻断。

## 8. 已有的安全基础

## 10. 需要特别避免的误解

### 误解 1：只要加上 TLS，这些问题就都没了

不是。

TLS 可以缓解窃听和中间人问题，但以下问题即使有 TLS 也依然值得修：

- 数据通道只靠 `AgentID` 握手。
- 会话没有真正绑定请求环境。
- token 放在 `localStorage`。
- 登录和 Agent 认证没有速率限制。
- 初始化接口可被抢占。
- ~~JWT Secret 有硬编码 Fallback。~~ ✅ 已修复
- HTTP Server 和 WebSocket 认证缺少超时保护。

### 误解 2：作为内网穿透工具，只要用户自己注意就行

不对。

既然产品明确支持无 TLS 使用，那么协议层就必须承担起基础安全责任。否则用户只是“能连上”，并不等于“能安全地使用”。
