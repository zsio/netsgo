## 当前代码观察

### 1. 控制台汇总口径仍然分散

当前仓库里已经有完整快照接口：

- `internal/server/console_api.go` 负责 `GET /api/console/snapshot`
- `internal/server/events.go` 会在 SSE 建连和定时同步时发送 `snapshot`

但前端概览数字仍然主要来自客户端列表的二次计算：

- `web/src/lib/console-summary.ts`
- `web/src/components/custom/layout/TopBar.tsx`
- `web/src/components/custom/client/ClientSidebar.tsx`

这意味着“页面展示数字”和“后端状态对象”还没有完全收口到同一个 authoritative contract。

### 2. `server.go` 仍然承担过多控制面职责

当前 `internal/server/server.go` 仍有约 1500 行，核心逻辑集中在：

- 服务启动 / 路由装配
- 控制连接鉴权
- `controlLoop`
- 隧道增删改停
- Web 面板相关 HTTP 处理

仓库已经开始做拆分，例如：

- `internal/server/console_api.go`
- `internal/server/events.go`
- `internal/server/server_lifecycle.go`

后续继续抽离时，应优先围绕“控制面消息处理”和“快照/状态整理”做同类职责聚合，避免把协议语义改动和文件拆分混在一次提交里。

### 3. 自动验证缺少显式 race lane

当前自动化入口仍然缺少一个稳定、可复用的 race 检查入口：

- `Makefile` 只有 `test`，没有单独的 `test-race`
- `.github/workflows/ci.yml` 的 Go 测试仍是 `go test ./...`

这会让并发问题的验证依赖人工记忆，而不是依赖固定流程。

## 本批次收口要求

### Issue #6：控制台汇总口径

建议收口标准：

- 后端提供一份统一的 console summary
- `server_status` / `console snapshot` / SSE `snapshot` 使用同一套汇总来源

- 后端测试与前端测试都要锁定“同一组数据 -> 同一组汇总数字”

### Issue #10：`server.go` 职责收缩

建议收口标准：

- `server.go` 只保留顶层装配和必要编排
- 控制面消息处理、快照整理等逻辑提取到同类文件
- 不改变当前 tunnel / session / auth 的语义
- 重构前后依赖测试证明行为一致

### Issue #12：并发验证补齐

建议收口标准：

- 本地有明确入口运行 `go test -race ./...`
- CI 有独立步骤或独立 job 执行 race 检查
- 若 race 检查暴露既有问题，应在结果中明确记录是“本次新引入”还是“仓库既有问题”
