# 🏗️ NetsGo 项目架构分析

> 面向当前代码实现的架构解读，重点回答“系统是如何组织的、优点在哪里、下一步该往哪里演进”。

## 1. 一句话结论

NetsGo 当前采用的是一个非常清晰的 **单仓库（Monorepo）+ 控制面/数据面分离 + Server/Agent 双端协同** 架构。  
从工程实现上看，它已经具备一个轻量级内网穿透平台的核心骨架，尤其适合继续向“可视化管控 + 持续隧道调度 + 节点监控”方向演进。

如果从专业架构师视角做评价：

- **优点很明显**：模块边界清晰、双端协议复用好、单端口设计有辨识度、扩展空间足够。
- **当前短板也明确**：安全能力、持久化能力、管理面能力还处于 Phase 1/2 早期阶段。
- **总体判断**：这是一个“架构方向正确、实现还在打基础”的项目，值得继续沿现有分层往下做，而不是推倒重来。

## 2. 当前总体架构

### 2.1 逻辑分层

项目可以拆成 5 个核心层次：

1. **入口层**
   - `/home/runner/work/netsgo/netsgo/cmd/server/main.go`
   - `/home/runner/work/netsgo/netsgo/cmd/client/main.go`
   - 负责参数解析、实例创建、生命周期启动。

2. **服务端核心层**
   - `/home/runner/work/netsgo/netsgo/internal/server/server.go`
   - `/home/runner/work/netsgo/netsgo/internal/server/data.go`
   - `/home/runner/work/netsgo/netsgo/internal/server/proxy.go`
   - 负责 Agent 接入、控制通道管理、数据通道管理、代理隧道调度。

3. **客户端核心层**
   - `/home/runner/work/netsgo/netsgo/internal/client/client.go`
   - `/home/runner/work/netsgo/netsgo/internal/client/probe.go`
   - 负责 Agent 认证、心跳、探针上报、数据通道建立、本地流量转发。

4. **共享协议层**
   - `/home/runner/work/netsgo/netsgo/pkg/protocol/message.go`
   - `/home/runner/work/netsgo/netsgo/pkg/protocol/types.go`
   - 负责双端共享的消息模型、探针结构、代理配置结构。

5. **共享传输能力层**
   - `/home/runner/work/netsgo/netsgo/pkg/mux/mux.go`
   - 负责基于 `yamux` 的多路复用封装与双向流量搬运。

### 2.2 核心设计思想

这个项目最重要的设计点不是“做了一个代理”，而是它把系统拆成了两个面：

- **控制面（Control Plane）**
  - 基于 WebSocket
  - 负责认证、心跳、探针上报、代理创建/关闭指令
  - 入口：`/ws/control`

- **数据面（Data Plane）**
  - 基于 TCP + 自定义握手 + `yamux`
  - 负责真实流量转发
  - 与 HTTP/WebSocket **共用一个端口**
  - 通过魔数 `0x4E` 识别数据通道连接

这意味着 NetsGo 的核心架构不是传统“一个后台 + 一个客户端”，而是 **一个统一接入端口下的多协议协同模型**。

## 3. 核心调用链路

### 3.1 Agent 上线链路

1. Client 从 `cmd/client/main.go` 启动。
2. `internal/client/client.go` 先连接 `ws://server/ws/control`。
3. Client 发送 `auth` 消息。
4. Server 在 `internal/server/server.go` 中完成认证并分配 `AgentID`。
5. Client 再通过同一主机端口建立原始 TCP 连接。
6. Client 发送数据通道握手：`[魔数][AgentID长度][AgentID]`。
7. Server 在 `internal/server/data.go` 中识别握手，创建 `yamux.Session`。
8. 之后控制消息和业务流量正式分离。

### 3.2 探针上报链路

1. Client 在 `internal/client/probe.go` 中采集 CPU、内存、磁盘、网络、运行时间。
2. 采样结果通过 `probe_report` 消息发往控制通道。
3. Server 在 `controlLoop` 中解析后更新内存态的 `AgentConn.Stats`。

### 3.3 代理转发链路

1. Client 或 Server 发起 `proxy_new` 请求。
2. Server 在 `internal/server/proxy.go` 中创建公网监听端口。
3. 外部访问到达该端口时，Server 在 Agent 的 `yamux` Session 上 `Open()` 一个 Stream。
4. Stream 头部写入 `proxy_name`，告知 Agent 这条流属于哪条代理。
5. Client 根据代理配置连接本地服务。
6. 双端通过 `pkg/mux.Relay` 进行双向拷贝。

## 4. 架构优点

### 4.1 模块边界清晰

`cmd`、`internal`、`pkg` 的职责划分比较规范，后续继续扩展时不容易失控：

- `cmd` 保持轻量，只做启动；
- `internal/server`、`internal/client` 承担业务编排；
- `pkg/protocol`、`pkg/mux` 承担跨端共享能力。

这对长期维护非常重要，因为它天然限制了“协议定义散落在双端代码里”的问题。

### 4.2 共享协议设计是正确方向

`pkg/protocol` 让 Server/Client 物理共享消息结构，是这个项目最有价值的工程决策之一。  
对于双端通信系统来说，这能显著降低字段漂移、版本错配、联调成本和灰度升级风险。

### 4.3 控制面 / 数据面分离合理

把“心跳、探针、管控指令”和“真实业务流量”拆开，是非常成熟的网络系统设计方式：

- 控制消息轻量、可观察、容易演进；
- 数据面可独立做吞吐优化；
- 后续接入鉴权、审计、限流时不会把所有逻辑耦合在同一条通道上。

### 4.4 单端口接入很有产品价值

Server 在 `internal/server/server.go` 里通过 peek 首字节区分 HTTP 请求和数据通道，这个设计很实用：

- 部署简单；
- 防火墙和运维开口成本低；
- 对“轻量级、开箱即用”的产品定位是加分项。

### 4.5 `yamux` 选型合理

`pkg/mux` 对 `yamux` 做了薄封装，没有过度抽象。  
这种做法对当前阶段是最优解：保留成熟库的能力，同时不把项目过早复杂化。

## 5. 当前架构短板与风险

### 5.1 Server 目前偏“内存编排器”

当前 Server 的 Agent、Stats、Proxy 状态主要在内存里维护。  
这让项目在原型期很轻，但也带来几个现实限制：

- 重启后状态丢失；
- 多实例扩展困难；
- 管理面无法沉淀历史数据；
- 审计和回放能力不足。

这不是架构错误，但说明它仍处于“单机、轻状态”的阶段。

### 5.2 认证与安全边界还比较薄

从当前实现看：

- `handleAuth` 阶段暂时接受所有连接；
- WebSocket `CheckOrigin` 直接返回 `true`；
- 还没有看到隧道级鉴权、Agent 身份校验、权限分层、审计日志。

这说明当前架构已经预留了控制面，但安全能力还没有真正落进去。  
如果后续面向生产环境，安全会是第一优先级演进方向。

### 5.3 Server 侧职责正在变重

目前服务端同时承担：

- Web 面板入口
- API
- Agent 接入
- 数据面接入
- 代理生命周期调度

在现阶段这没有问题，但后续如果继续加 SQLite、Web CRUD、审计、监控大屏，`internal/server` 会逐步变成“大包”。  
建议未来在不改变总体架构的前提下，继续做子模块拆分，例如：

- `server/control`
- `server/data`
- `server/proxy`
- `server/api`
- `server/store`

### 5.4 观测性还不足

虽然项目本身已经有“探针监控”概念，但更多是业务功能视角。  
从系统运维视角看，目前仍缺少：

- 结构化日志；
- 关键链路指标；
- 连接/流量/失败原因的统一观测面；
- 对控制面和数据面的排障工具。

## 6. 演进建议

如果从架构治理角度给出下一步建议，我会按优先级分三层推进：

### 第一优先级：把生产可用性补齐

1. **完善 Agent 认证**
   - 至少引入 Token 校验；
   - 数据通道建议绑定一次性校验信息或短期凭证。

2. **引入持久化层**
   - 先从 SQLite 开始；
   - 落配置、Agent 元数据、代理规则、最近探针快照。

3. **补审计与日志**
   - 记录 Agent 上下线、代理创建/关闭、端口绑定失败、握手失败等关键事件。

### 第二优先级：把 Server 做成真正的管控中心

1. 增加 Web/API 的资源模型；
2. 把 Agent、Proxy、Stats 统一抽象为可查询对象；
3. 为 Web 面板提供稳定的数据读取接口。

### 第三优先级：为高阶能力留扩展点

1. 多 Server 级联；
2. 更丰富的协议支持（UDP/HTTP 路由）；
3. 更细粒度的流量治理、审计和告警；
4. 更强的终端运维能力（如 Web Terminal、批量运维、服务发现）。

## 7. 专业架构师结论

从项目现状来看，NetsGo 的架构基础是 **健康的、可延展的、工程上顺手的**。

它最值得肯定的地方有三点：

1. **双端共享协议**，避免了很多穿透类项目后期最痛苦的联调问题；
2. **控制面和数据面分离**，为未来演进预留了足够空间；
3. **单端口 + 单文件交付思路清晰**，非常符合轻量平台产品定位。

如果后续持续投入，我建议坚持现在这条主线：

> **保持 Monorepo + 共享协议 + 控制面/数据面分离不变，优先补安全、持久化、观测性，再逐步做管理面增强。**

这条路线最稳，也最符合项目当前代码基础。
