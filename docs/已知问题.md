# 已知问题

## 🔴 P0 — 数据通道无认证（安全漏洞）

**状态**: 待修复（计划在添加认证 key 时一并修复）

**问题描述**:

当前数据通道握手仅发送 AgentID，而 AgentID 格式为 `agent_主机名_序号`，容易被猜测。攻击者可以直接向 Server 发送 `0x4E + AgentID` 来劫持 Agent 的数据通道。

**涉及文件**:
- `internal/server/data.go` — `handleDataConn()` 只做 AgentID 查找，无鉴权
- `internal/client/client.go` — `connectDataChannel()` 握手只发 AgentID

**修复方案**:

在添加认证 key 时，同步实现以下逻辑：

1. Server 在 `auth_resp` 中返回一个 **一次性随机 data_token**（`crypto/rand` 生成 32 字节）
2. Agent 用 `data_token`（而非 AgentID）进行数据通道握手
3. Server 验证 token 后找到对应 Agent，建立 yamux Session
4. Token 验证成功后立即失效（一次性使用）

```
auth_resp 新增字段:
  DataToken string `json:"data_token,omitempty"`

数据通道握手变更:
  旧: 0x4E + [2B AgentID长度] + [NB AgentID]
  新: 0x4E + [2B Token长度]   + [NB DataToken]
```

## 🟡 P1 — Agent ID 自增可预测

**状态**: 待修复（随认证 key 一起改进）

**问题描述**: `generateID()` 使用自增计数器，重启后归零产生重复 ID，且格式可预测。

**涉及文件**: `internal/server/server.go` L367-375

**修复方案**: 改用 `crypto/rand` 或 UUID。

## 🟡 P1 — agent.Stats 读写无锁保护

**状态**: 待修复

**问题描述**: `controlLoop` 写 `agent.Stats` 和 `handleAPIAgents` 读 `agent.Stats` 在不同 goroutine 中无锁访问，存在 data race。

**涉及文件**: `internal/server/server.go` L260, L356

**修复方案**: 使用 `atomic.Pointer[protocol.SystemStats]` 或在 `AgentConn.mu` 保护下读写。

## 🟡 P1 — PeekListener pending channel 满时阻塞

**状态**: 待修复

**问题描述**: `pending` channel 缓冲为 64，满时 `dispatchLoop` 阻塞，导致数据通道连接也无法被 accept。

**涉及文件**: `internal/server/server.go` L110, L144

**修复方案**: 使用 `select + default` 溢出保护，或增大缓冲。

## 🟢 P2 — Server 无 Graceful Shutdown

**状态**: 待实现

**问题描述**: `Server.Start()` 无关闭方法，进程只能通过 kill 退出。Agent 断开后代理隧道的 Listener 可能未及时清理。

**修复方案**: 保存 `net.Listener` 和 `http.Server` 引用，提供 `Stop()` 方法；`main.go` 监听 `os.Signal`。
