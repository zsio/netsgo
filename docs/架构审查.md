# NetsGo 架构审查

审查时间：2026-03-10

## 1. 审查结论

从整体方向看，这个项目的架构思路**是合理的**，尤其适合当前 Phase 1 / MVP 阶段：

- 使用 `cmd / internal / pkg / web` 的目录分层，职责边界清晰；
- 服务端与客户端共享 `pkg/protocol`，可以显著降低双端联调成本；
- “控制面（WebSocket）+ 数据面（TCP + yamux）”的拆分是正确方向；
- 单端口接入的设计，能够降低部署门槛，符合“轻量级工具”的产品定位。

但从当前代码实现来看，项目仍然处于**架构正确、工程完成度偏早期**的阶段：核心主链路已经跑通，但在**安全性、可靠性、可运维性、文档一致性**四个方面存在明显短板。  
因此，当前架构更适合定义为：**一个方向正确的 MVP 骨架，而不是已经稳态可扩展的生产级架构。**

---

## 2. 当前架构的合理之处

### 2.1 分层结构基本合理

当前仓库结构：

```text
cmd/
internal/
pkg/
web/
docs/
```

这一层次划分符合 Go 项目的常见工程组织方式：

- `cmd/server`、`cmd/client`：可执行入口；
- `internal/server`、`internal/client`：服务端/客户端内部实现；
- `pkg/protocol`、`pkg/mux`：可复用的共享能力；
- `web`：前端独立工程；
- `docs`：设计、路线图、已知问题。

这个结构对后续继续演进是友好的，没有出现“所有逻辑全塞到 main 包”或者“协议、业务、传输层强耦合到一起”的问题。

### 2.2 控制面与数据面分离是正确的

从实现看：

- 控制面：`/ws/control`，承载认证、心跳、探针上报、代理指令；
- 数据面：同端口 TCP 二进制握手，之后交给 `yamux` 做多路复用。

相关实现：

- `/home/runner/work/netsgo/netsgo/internal/server/server.go`
- `/home/runner/work/netsgo/netsgo/internal/server/data.go`
- `/home/runner/work/netsgo/netsgo/internal/client/client.go`

这个设计比“所有控制与数据都走同一个 WebSocket JSON 通道”更合理，优点是：

- 控制信令轻量化；
- 数据通道可独立优化；
- `yamux` 复用现成成熟能力，避免自研协议复杂度。

### 2.3 Monorepo + 共享协议定义是加分项

`pkg/protocol` 统一承载 `AgentInfo`、`SystemStats`、消息类型、代理配置等定义。  
对于这种双端必须强一致的系统，这是非常合适的做法，能够有效降低：

- JSON 字段漂移；
- 双端协议升级不同步；
- 测试用例难以复用的问题。

这块设计是本项目目前最值得保留的核心资产之一。

## 3. 主要架构缺陷与风险

## 3.1 安全边界过弱，当前更像“开发态联通”而非“生产态可用”

这是目前最明显的问题。

### 问题 A：控制通道认证实际上未生效

`handleAuth()` 当前“接受所有连接”：

- `/home/runner/work/netsgo/netsgo/internal/server/server.go`

这意味着：

- Token 目前只是协议字段，不是实际安全边界；
- 任意客户端都可以注册为 Agent；
- 后续所有代理能力都建立在一个不可信前提上。

### 问题 B：数据通道仅依赖可预测的 AgentID

数据通道握手当前为：

```text
0x4E + AgentID
```

相关实现：

- `/home/runner/work/netsgo/netsgo/internal/server/data.go`
- `/home/runner/work/netsgo/netsgo/internal/client/client.go`

而 `AgentID` 又是可预测格式：

```text
agent_<hostname>_<自增序号>
```

这会导致数据通道很容易被伪造或劫持。  
这一点与 `/docs/已知问题.md` 中列出的 P0 问题一致，属于必须优先修复的架构缺陷。

### 问题 C：默认允许所有 WebSocket Origin

`websocket.Upgrader.CheckOrigin` 直接返回 `true`：

- `/home/runner/work/netsgo/netsgo/internal/server/server.go`

在开发阶段可以接受，但如果未来接 Web 管理台或公网部署，这会放大跨站调用风险。  
如果后续面板具备登录态或管理能力，这里必须收紧。

### 建议

短期至少应补齐：

1. 控制通道 token 校验；
2. 数据通道改为一次性随机 token，而不是 AgentID；
3. 明确 TLS/WSS 方案；
4. 前后端管理接口与 Agent 接入接口分离鉴权策略。

## 3.2 架构文档与实际实现存在偏差

项目的“设想”是好的，但当前实现和文档之间有几个明显偏差，会影响团队协作判断。

### 偏差 A：README 对数据通道描述与实际代码不完全一致

README 架构图中写的是：

```text
/ws/data → 数据通道 (WebSocket)
```

但实际代码中：

- `/ws/data` 只是返回说明信息；
- 真正的数据通道已经迁移为**同端口 TCP 二进制协议**。

这类文档漂移会让后来维护者误解真实架构。

### 偏差 B：README 仍强调 `web/dist`，但服务端尚未真正嵌入前端

当前仓库已经有 `web/` 工程，但服务端代码还没有完成 `go:embed` 的前端集成闭环。  
也就是说，“单文件交付”目前仍然主要停留在架构目标，而不是完整落地状态。

### 偏差 C：设计文档中提到 SQLite，本期实现中尚未出现

`/docs/设想和方案.md` 中明确把 SQLite 作为服务端核心能力之一，但当前服务端状态仍然是纯内存：

- Agent 用 `sync.Map` 保存；
- Proxy 信息保存在内存 map 中；
- 进程重启后状态丢失。

这不是错误，但应该明确标注为“路线图未完成”，否则会误导对系统能力的判断。

---

## 3.3 可靠性方面仍有几个典型并发/生命周期问题

这部分问题会直接影响线上稳定性。

### 问题 A：缺少优雅关闭能力

`Server.Start()` 直接 `Serve()`，没有显式 `Stop()` / graceful shutdown：

- `/home/runner/work/netsgo/netsgo/internal/server/server.go`

影响：

- 进程退出时连接和 listener 的回收不完整；
- 代理 listener 生命周期难统一管理；
- 后续做 systemd / 容器部署时，滚动升级体验会较差。

### 问题 B：Agent 状态并发读写保护不足

`agent.Stats` 在控制循环里写，在 API 里读，当前没有统一同步约束。  
这个问题已经被记录在：

- `/home/runner/work/netsgo/netsgo/docs/已知问题.md`

对 Go 服务来说，这种共享状态 race 很常见，也很容易在压测或 `-race` 下暴露。

### 问题 C：代理创建存在 TOCTOU 窗口

`StartProxy()` 中“检查是否存在”与“插入 map”之间存在时间窗口。  
这同样已经被现有文档识别为已知问题：

- `/home/runner/work/netsgo/netsgo/internal/server/proxy.go`
- `/home/runner/work/netsgo/netsgo/docs/已知问题.md`

说明团队对问题有认知，但目前尚未完成工程收敛。

### 问题 D：PeekListener 的背压策略较弱

`PeekListener` 使用固定大小 `pending` channel。  
当 HTTP 连接积压到一定程度时，分发循环可能阻塞，进而影响整体入口处理能力：

- `/home/runner/work/netsgo/netsgo/internal/server/server.go`

对单端口多协议模型来说，入口 listener 是关键共享资源，这里需要更明确的限流/退避策略。

---

## 3.4 可扩展性目前主要依赖“单机内存状态”，后续扩容会比较吃力

当前架构适合单机 MVP，但离多节点部署还有明显距离。

### 现状

- Agent 注册状态在内存；
- Proxy 生命周期在内存；
- 没有统一配置中心；
- 没有持久化事件流；
- 没有集群路由层设计。

### 影响

后续如果要支持：

- 多实例部署；
- 管理台横向扩展；
- Server 重启恢复；
- 代理规则持久保存；

就必须补上至少一层持久化和状态恢复机制。  
所以当前设计**不是错误**，但它的适用边界应该明确限定在“单机 MVP / 验证阶段”。

---

## 3.5 可运维性仍偏弱

从架构视角看，一个网络基础设施项目除了能跑，还要便于排查问题。当前这部分能力偏弱：

- 日志以 `log.Printf` 为主，尚未形成结构化字段；
- 没有统一 trace / metrics；
- 缺少“控制通道是否健康、数据通道是否重连成功、代理转发成功率”等可观测性指标；
- 前端工程存在，但服务端没有形成完整交付闭环。

这意味着一旦线上出现“能连上但偶发转发失败”的问题，当前排查成本会偏高。

---

## 4. 架构成熟度判断

如果按成熟度划分，我会这样评价：

- **架构方向**：7.5 / 10  
- **MVP 合理性**：8 / 10  
- **生产可用性**：4 / 10  
- **安全完备度**：3 / 10  
- **工程闭环完整度**：5 / 10

换句话说：

> 这不是一个“架构有根本性问题”的项目；  
> 这是一个“架构方向正确，但从提案走到生产落地，中间还有几个关键台阶没补齐”的项目。

---

## 5. 建议的优先级路线

### P0：先补安全和可靠性底线

建议优先完成：

1. 控制通道鉴权真正落地；
2. 数据通道从 AgentID 握手切换到一次性 token；
3. 修复 `agent.Stats` 并发读写；
4. 修复 `StartProxy()` 的 TOCTOU；
5. 增加服务端优雅关闭能力。

### P1：让“架构目标”和“工程现实”对齐

1. 修正文档中对数据通道的描述；
2. 明确哪些能力已经实现，哪些还在路线图中；
3. 完成 `web` 到服务端交付链路的闭环（构建、嵌入、发布）。

### P2：再做平台化演进

1. 引入持久化（如 SQLite）；
2. 增加结构化日志与基础监控指标；
3. 评估未来多节点/高可用的状态模型；
4. 再继续推进监控大盘、限速、HTTP 路由等增强能力。

---

## 6. 最终判断

**结论一句话：这个项目的架构“方向是对的”，但当前实现更像一个优秀的原型，而不是完整的生产架构。**

如果目标是：

- 先快速验证产品价值；
- 验证单端口 + 探针 + 隧道一体化方向；
- 给后续平台化能力打骨架；

那么当前架构是成立的。

但如果目标是：

- 面向公网长期运行；
- 面向多人协作和正式运维；
- 面向可恢复、可审计、可扩展的生产环境；

那么当前还需要优先补上：**认证、安全、并发一致性、生命周期管理、持久化** 这五个关键台阶。
