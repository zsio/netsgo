# NetsGo 安全与传输架构实施计划（统一口径版）

## 1. 文档目的

本文用于把以下三类文档串起来，形成统一的研发落地口径：

1. `目前存在的问题.md`：定义当前风险与边界。
2. `P1解决方案一_设计文档.md`：定义目标架构与架构约束。
3. 本文：定义实施阶段、任务拆解、验收顺序与协作边界。

本文不重复论证“为什么这是问题”，而是聚焦：

- 先做什么；
- 后做什么；
- 哪些能力要并行验证；
- 哪些问题不能在实施过程中被误判为“已顺带解决”。

## 2. 当前实现与目标态

### 2.1 当前实现（必须统一认知）

- 控制通道：`ws://.../ws/control`（支持 `wss://`，需通过 `--tls-mode` 启用）
- 数据通道：`PeekListener + 0x4E + 裸 TCP + yamux`（TLS 模式下自动走 TLS 层）
- **阶段 1 已落地**：
  - `tls.mode = custom/auto/off` 三态配置
  - 服务端 TLS Listener 包装
  - 客户端 `wss://` 控制连接 + TLS 数据通道拨号
  - TOFU 证书指纹校验与持久化
  - HSTS 安全头自动注入
  - 地址模型统一（`ws/wss/http/https` 四格式兼容）
- 当前代码尚未落地：
  - 真实 `/ws/data` WebSocket 数据面

### 2.2 目标态

- Web 面板：`https://host/`
- API：`https://host/api/...`
- 控制通道：`wss://host/ws/control`
- 数据通道：`wss://host/ws/data`
- 过渡期支持旧数据面与新数据面双栈并存

### 2.3 地址模型（新增，实施必须统一）

P1 的实施必须统一“服务地址”的语义，避免前端、CLI、后端存储和文档各自采用不同模型。

统一定义如下：

1. `server_addr` 表示**公共 HTTP 基地址**，例如：
   - `http://127.0.0.1:8080`
   - `https://tunnel.example.com`
2. 控制通道地址由 `server_addr` 推导：
   - `http://` → `ws://.../ws/control`
   - `https://` → `wss://.../ws/control`
3. 数据通道地址由 `server_addr` 推导：
   - `http://` → `ws://.../ws/data`
   - `https://` → `wss://.../ws/data`
4. 当前版本尚未完成这个统一模型，因此阶段 0 与阶段 1 必须同步处理：
   - 前端初始化地址提示；
   - 后台保存的 `server_addr`；
   - 生成给用户的 Agent 接入命令；
   - 客户端参数语义与帮助文本。

## 3. 总体实施原则

### 3.1 先止血，再收敛，再清理

按顺序推进：

1. 先补 TLS 基础能力，尽快降低明文风险；
2. 再做 `/ws/data` 数据面收敛；
3. 再做双栈迁移、灰度和旧协议清退；
4. 最后收口浏览器侧和前端敏感数据暴露问题。

### 3.2 不允许“文档先领先代码一个宇宙”

每个阶段完成后，必须同步更新：

- `README.md`
- `目前存在的问题.md`
- `P1解决方案一_设计文档.md`
- 本文

避免再次出现“README 以为 `/ws/data` 已上线，但代码其实还没动”的情况。

### 3.3 不把 TLS 当万能胶

实施过程中必须持续提醒：

- TLS 主要解决传输机密性与完整性；
- 不自动解决 `localStorage` token 风险；
- 不自动解决前端敏感材料回传风险；
- 不自动替代所有业务级防重放需求。

### 3.4 不把阶段 1 当终态

实施过程中必须统一说明：

1. 阶段 1 是“TLS 止血”，不是最终架构完成。
2. 只要数据面仍停留在 `PeekListener + 0x4E + yamux`，长期接入模型就还没有真正收敛。
3. 阶段 2 的 `/ws/data` 不属于可无限后移的“优化项”，而是 P1 长期闭环的一部分。

## 4. 分阶段计划

## 阶段 0：文档校准与配置面预留

### 阶段 0 目标

统一团队认知，并为后续 TLS / 新数据面预留配置形态。

### 阶段 0 主要任务

1. 文档统一口径：现状 vs 目标态。
2. 设计并冻结配置键：
   - `tls.mode`
   - `tls.cert_file`
   - `tls.key_file`
   - `tls.trusted_proxies`
   - 是否保留旧数据面兼容开关
3. 冻结地址模型：
   - `server_addr` 的唯一语义；
   - `http/https` 到 `ws/wss` 的映射规则；
   - 前端与客户端使用同一套推导逻辑。
4. 明确日志与观测字段命名。

### 阶段 0 出口条件 ✅ 已达成

- ✅ 团队对"当前真实架构"和"目标架构"无歧义；
- ✅ 配置面约束完成（`TLSConfig` 结构体已冻结，CLI flags 已实现）；
- ✅ 地址模型已冻结（`normalizeServerAddr` + `deriveControlURL` 已实现）。

---

## 阶段 1：TLS 基础能力（止血）

### 阶段 1 目标

先让控制链路进入可加密状态，尽快降低 P1 风险。

### 阶段 1 服务端任务

1. 增加 TLS 配置加载能力。
2. 实现 `tls.mode = custom`。
3. 实现 `tls.mode = auto`：
   - 自动生成自签名证书；
   - 证书或根证书持久化；
   - 支持 IP SAN / Hostname SAN。
4. 实现 `tls.mode = off`：
   - 增加 `trusted_proxies`；
   - 仅对 trusted proxy 来源解析 `X-Forwarded-*`。
5. 明确阶段 1 的过渡实现边界，并在日志/文档中标明其非终态属性。

### 阶段 1 客户端任务

1. 支持 `wss://` 控制连接。
2. 支持正式证书标准校验。
3. 支持 `auto` 模式下 TOFU / pinning：
   - 首次记录指纹；
   - 后续严格比对；
   - pin mismatch 默认拒绝。
4. 支持从 `server_addr` 的 `http/https` 语义统一推导控制通道地址。
5. **数据通道 TLS 对接**（架构审查补充）：
   - 当前数据通道使用 `net.DialTimeout("tcp", ...)` 建立裸 TCP 连接；
   - 单端口架构下 Server 加 TLS 等于全端口 TLS，数据通道也经过 TLS 层；
   - 客户端必须根据 `server_addr` 协议选择 `net.Dial` 或 `tls.Dial`。
6. **`--server` 参数向后兼容**（架构审查补充）：
   - 同时接受 `ws://`、`wss://`、`http://`、`https://` 四种输入格式；
   - 内部统一规范化为 HTTP 基地址后再推导；
   - 默认值暂保持 `ws://localhost:8080`，后续版本再切换为 `http://`。

### 阶段 1 前端与产品任务

1. 初始化页继续使用完整公共地址输入，但需与客户端地址推导规则一致。
2. 后台生成的 Agent 接入命令必须与真实 CLI 参数语义一致。
3. 接入提示文案需明确当前阶段是“TLS 止血”而非最终数据面收敛完成。

### 阶段 1 测试任务

1. 自签证书重启后指纹稳定性测试。
2. 标准证书链校验测试。
3. trusted proxy 信任边界测试。
4. 不同部署模式下控制面连通性测试。
5. `server_addr` 到控制通道地址的推导测试。
6. 前端生成的接入命令与客户端实际参数兼容性测试。

### 阶段 1 已知限制（架构审查补充）

1. **`tls.mode=off` + 反向代理场景下，旧数据面不可用**：
   - 七层反代（Nginx/Caddy）无法转发首字节 `0x4E` 的裸 TCP 二进制协议；
   - 该场景下控制面可通过 WebSocket 正常工作，但数据面需等阶段 2（`/ws/data`）才能完整支持；
   - 此为已知限制，不阻塞阶段 1 交付。

### 阶段 1 出口条件

- 控制通道可稳定在 TLS 下工作；
- **数据通道在 `custom`/`auto` 模式下可稳定在 TLS 下工作**；
- 默认部署不再只剩明文可选；
- `auto` 模式的证书信任模型稳定可复现；
- 地址模型已在后端、前端、客户端三处对齐；
- 阶段 1 在文档、产品提示和实现层都未被错误标注为终态。

---

## 阶段 2：`/ws/data` 新数据面实现（收敛）

### 阶段 2 目标

完成从裸 TCP 数据面到 WebSocket 数据面的能力建设。

### 阶段 2 服务端任务

1. 提供真实 `/ws/data` 端点。
2. 实现 WS ↔ 字节流适配层。
3. 在 `/ws/data` 上承载 `yamux`。
4. 增加数据面连接日志、错误码与观测指标。

### 阶段 2 客户端任务

1. 支持通过 `/ws/data` 建立新数据通道。
2. 支持与控制通道协同建立数据面。
3. 支持超时、关闭与重连处理。
4. 继续沿用统一地址模型推导 `/ws/data` 连接地址。

### 阶段 2 适配层专项要求

1. 正确处理 WebSocket 消息边界与字节流语义差异。
2. 正确处理并发读写约束。
3. 正确处理 close、EOF、deadline、keepalive。
4. 为后续压测预留缓冲与观测点。

### 阶段 2 测试任务

1. TCP 隧道功能回归。
2. 长连接稳定性测试。
3. 大包/小包混合流量测试。
4. 反代后 `/ws/data` 连通性测试。

### 阶段 2 出口条件

- `/ws/data` 在主要 TCP 场景可稳定工作；
- 服务端与客户端对新数据面有完整日志与指标；
- 反代场景可正常接入。

---

## 阶段 3：双栈兼容、灰度与回滚

### 阶段 3 目标

避免旧 Agent 大面积掉线，降低替换风险。

### 阶段 3 服务端任务

1. 同时保留：
   - 旧数据面：`PeekListener + 0x4E + yamux`
   - 新数据面：`/ws/data + WSS + yamux`
2. 支持配置选择优先协议。
3. 输出新旧协议使用比例与失败率指标。

### 阶段 3 客户端任务

1. 新版本优先尝试 `/ws/data`。
2. 在兼容窗口期内，必要时支持回退旧协议。
3. 记录回退原因，便于排障。

### 阶段 3 测试任务

1. 新旧版本混跑测试。
2. 新旧协议切换测试。
3. 配置回滚测试。
4. 异常网络环境下回退策略测试。

### 阶段 3 出口条件

- 新旧协议共存稳定；
- 升级不会导致旧 Agent 即刻全量失效；
- 回滚开关可用且已验证。

---

## 阶段 4：旧协议清退与浏览器侧风险收口

### 阶段 4 目标

完成架构收敛，并补齐 TLS 不解决的残余高风险项。

### 阶段 4 主要任务

1. 评估并决定旧数据面清退窗口。
2. 处理 P5：
   - 管理端 token 从 `localStorage` 迁移到更稳妥的会话模型。
3. 处理 P11：
   - 缩短敏感材料在浏览器端暴露窗口；
   - 优化一次性展示与确认流程。
4. 更新对外部署文档与升级说明。

### 阶段 4 出口条件

- 项目对外接入形态完成统一；
- P5 / P11 有独立整改结论；
- 文档、实现、部署说明再次保持一致。

## 6. 统一沟通口径

团队内部或对外说明时，统一使用以下口径：

1. **当前实现**：控制面是 `ws://`，数据面仍是同端口二进制协议。
2. **目标架构**：统一到 `https://` / `wss://`，并引入真实 `/ws/data`。
3. **TLS 的作用**：解决传输安全问题，不自动解决浏览器会话和前端敏感信息暴露问题。
4. **地址模型**：`server_addr` 表示公共 HTTP 基地址，控制/数据通道地址由其推导得到。
5. **TOFU 的作用**：改善自签名场景的后续信任，不解决首次连接信任建立问题。
6. **迁移策略**：采用双栈兼容与可回滚切换，不做一次性硬切换。

## 7. 实施完成定义

只有同时满足以下条件，才能认为本轮架构整改真正完成：

1. 文档、代码、部署说明三者一致；
2. 控制面和数据面都已具备稳定 TLS 能力；
3. `/ws/data` 已在主要场景完成验证；
4. 旧协议清退或兼容策略有明确结论；
5. P5 / P11 没有被“被 TLS 顺带解决”的错误叙事覆盖掉；
6. `server_addr`、前端生成命令、客户端参数语义与实际实现完全一致。
