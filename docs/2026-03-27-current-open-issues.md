# 当前问题与状态说明（2026-03-27）

## 1. 文档说明

这份文档是当前仓库里唯一保留的状态说明和问题清单。

它的用途有两个：

1. 说明当前系统里几类状态分别是什么
2. 列出当前还没有处理完的问题，方便后续逐项推进

这份文档已经吸收了之前审查文档里仍然有效的内容，因此后续只维护这一份，不再并行维护第二份同类文档。

需要先说明一个边界：

- **隧道状态**，只描述隧道本身的情况
- **本地服务状态**，不属于当前隧道状态的一部分，后续如果要做，会单独规划

也就是说，不能再把“本地服务是否健康”直接当成“隧道是否健康”的定义内容。

## 2. 先说结论

当前项目已经具备核心能力，服务端、客户端、管理页面、隧道创建和恢复这些主流程已经能工作。

而且按当前实现，系统里的状态数据本来就是独立于页面存在的：

- 客户端会主动上报自己的机器状态
- 服务端会自己维护当前进程里已知的最新状态
- 页面只是读取和展示服务端已经掌握的数据

所以当前真正需要解决的，不是“页面没打开就不知道状态”，而是下面这些更具体的问题：

1. 有些重要数据写文件失败时，接口仍可能返回成功
2. 某些异常情况下，系统会带着空数据继续运行
3. 某些真实失败没有及时、完整地反映到状态和页面上
4. 页面重连后已经可以主动拉一次完整状态，但慢连接纠偏目前仍保留“定时再拿一次完整状态”这条安全网
5. 有些状态看起来有值，但没有清楚说明这是“什么时候的数据”

## 3. 当前系统里几类状态分别是什么

### 3.1 服务端自身状态

这类状态描述的是服务端自己当前的情况，例如：

- 是否正在运行
- 监听端口
- 自己的 CPU、内存、磁盘使用情况
- 当前连接数量
- 当前隧道数量

这类状态由服务端自己定时整理，和页面是否在线没有关系。

### 3.2 客户端机器状态

这类状态描述的是客户端所在机器的情况，例如：

- CPU
- 内存
- 磁盘
- 公网 IP
- 运行时长

按当前实现，这些数据不是页面临时去“要”的，而是：

- 客户端启动后先主动上报一次
- 之后每 10 秒继续上报一次
- 服务端收到后更新自己掌握的数据

所以这部分状态本来就应该独立于页面存在。

### 3.3 隧道运行状态

这类状态只描述隧道本身，例如：

- 配置是否存在
- 当前是运行、暂停、停止还是出错
- 入口是否已经建立
- 当前有没有明确的错误信息

这里要特别强调：

> **隧道运行状态，不等于本地服务状态。**

同时还要明确一件事：

> **当前创建成功、ready、exposed，只表示配置已被接受、入口已建立，不表示本地服务已验证可连，也不表示第一笔真实流量一定成功。**

这也意味着，当前不会把下面这些内容直接写进“隧道健康”里：

- 客户端连接本地服务是否成功
- 第一笔真实流量是否成功
- 最近成功率是多少

当前文档里的隧道状态，只关心隧道这套链路本身是否还能继续工作，例如：

- TCP 监听是否还正常
- UDP 主读循环是否还正常
- 服务端到客户端的转发通道是否还能打开
- 长时间运行时，连接、会话、清理是否还正常

如果后续确实需要观察本地服务可用性或真实流量结果，应单独规划，不混进当前隧道状态。

### 3.4 本地服务状态（未来单独能力）

本地服务状态指的是：

- 本地服务是否真的启动了
- 本地端口后的业务是否真的可以正常响应
- 本地服务是否长期稳定

这类能力后续如果要做，应该单独设计，至少要先明确：

- 是否由用户显式开启
- 用什么方式检查
- 检查频率是多少
- 检查结果怎么展示

在这套能力明确之前，不应该把它混进当前隧道状态定义里。

## 4. 当前状态数据是怎样流动的

按当前实现，状态数据大致是这样流动的：

### 4.1 客户端主动上报机器状态

客户端启动后，会先上报一次机器状态，之后每 10 秒再上报一次。

这一步和页面在不在线没有关系。

### 4.2 服务端接收并维护状态

服务端收到客户端上报后，会做几件事：

- 更新当前内存里的状态
- 尽力把最近一次客户端状态写入存储
- 在有在线页面时，把变化通知发出去

这里也要说清楚一个边界：

> **服务端维护的是“当前进程里已知的最新状态”，并尽力把最近状态写入存储，但这不是绝对不会丢的保证。**

例如写文件失败时，现在有些地方仍然只是打日志。

### 4.3 页面只负责读取和展示

当前页面主要做两件事：

- 读取服务端给出的当前状态
- 接收服务端推送过来的变化通知

页面不应该承担下面这些职责：

- 决定客户端什么时候去采集 CPU、IP 等机器数据
- 自己推断“本地服务一定健康”
- 把不完整的数据再解释成另一层状态

### 4.4 现在的页面同步方式

当前页面同步状态的方式是：

- 平时接收“变化通知”
- 同时，服务端还会定时再发一次完整状态，用来把页面纠正回来

这样做的原因很直接：

- 如果页面处理不过来，某些变化通知会丢
- 一旦丢了，页面自己现在还不能可靠地知道“我漏掉了什么”
- 所以系统还需要靠“定时再发一次完整状态”来修正显示

这套方式现在能工作，但后面还需要继续整理。

## 6. 需要优先处理的问题

### 6.1 管理数据写文件失败时，部分接口仍可能返回成功

当前有一部分管理数据修改流程，会先改内存，再写文件。

问题在于，某些地方写文件失败后，并没有把失败继续往上返回，所以会出现这种情况：

- 接口已经返回成功
- 内存里的数据看起来也改了
- 但文件其实没有写成功

最直接的影响包括：

- 登录会话
- API key
- 客户端 token
- 最近活跃时间

这类问题最容易让人误判，因为当时看起来一切都正常，重启后才发现数据没有保存下来。

### 6.2 配置文件损坏时，系统可能用空数据继续运行

当前对配置文件损坏的处理还不够稳妥。

在部分读取失败场景里，系统会：

- 记录日志
- 把内部数据改成空
- 然后继续启动

这样会导致：

- 隧道列表突然变空
- 管理员、客户端、token 等信息看起来像丢失了
- 实际根因是配置文件坏了，但外部看起来像普通业务问题

这里要说清楚，这个问题主要指的是：

- 文件损坏
- JSON 解析失败

并不是所有启动异常都会继续运行。

### 6.3 隧道状态和本地服务状态的边界，后续还要继续保持清楚

这里要明确一个后续方向：

- 隧道状态只描述隧道本身
- 本地服务可用性不放进当前隧道状态
- 当前也不计划单独做“最近成功率”这种指标

因此，下面这些内容不应直接写成当前隧道健康的一部分：

- 客户端连接本地服务是否成功
- 第一笔真实流量是否成功
- 最近几分钟成功率是多少

当前文档里的隧道健康，只关注隧道这条链路本身是否还能继续工作，例如：

- TCP 监听是否还正常
- UDP 主读循环是否还正常
- 服务端到客户端的转发通道是否还能打开
- 长时间运行时，连接、会话、清理是否还正常

如果后续确实需要观察本地服务可用性或真实流量结果，应单独规划，不混进这里。

### 6.4 某些隧道失败场景没有把最终结果完整通知到页面

当前页面收到的通知，不是每一次都能完整反映“开始处理”和“最终结果”。

已经确认的一条路径是：

- 创建隧道时，页面先会收到“处理中”
- 如果后面等待确认超时，接口会返回失败
- 但在已观察到的时间窗口里，没有收到这条隧道对应的最终失败通知

这会导致：

- 页面在一段时间内显示不准
- 还要等后面的完整状态把它纠正回来

### 6.5 当前状态缺少清楚的时间信息，页面不容易判断这些数据到底是什么时候的

现在很多状态值虽然能看到，但“它是什么时候的”并不总是清楚。

这会带来几个很实际的问题：

- 客户端显示在线，但机器状态可能已经不是刚刚上报的
- 服务端状态是缓存值，但页面上看不出它是什么时间整理出来的
- 页面重新拿到一份完整状态后，也不知道这是哪一刻生成的

如果不把时间信息补清楚，后面状态越多，越容易让人误解。

## 7. 建议尽快处理的问题

### 7.1 页面展示和后台统计使用的计算方式不同，数字可能对不上

当前页面上的客户端和隧道列表，与后台状态统计，不是完全从同一套数据路径算出来的。

这意味着：

- 某些页面看到的数量
- 某个状态栏里显示的数量
- 某些接口返回的汇总数字

在短时间内可能对不上。

这不一定马上造成故障，但会增加排查难度。

### 7.2 实时推送对慢连接不够友好，目前仍依赖定时发送完整状态来纠正页面

当前页面同步方式可以概括成两部分：

- 平时靠变化通知更新
- 定时再发一次完整状态，把页面纠正回来

这套方式之所以还不能直接取消后一部分，是因为：

- 慢连接时，变化通知会丢
- 页面自己现在还不能可靠地发现“我漏了哪些变化”

所以后续更合理的方向不是“直接停掉定时完整状态”，而是分两步走：

#### 第一步
先保留当前这套方式，同时把状态口径和时间信息写清楚。

#### 第二步
这一部分已经开始落地：现在已有 `GET /api/console/snapshot`，主要用于：

- 页面第一次进入时
- 页面重新连上之后
- 用户手动刷新时
- 将来系统有能力判断页面状态可能不可信时

当前实现里，页面重连成功后会先主动同步一次完整快照，但只有在页面和服务端都能更可靠地知道“变化有没有漏掉”之后，才适合再考虑去掉当前的定时完整状态。

### 7.3 客户端重连时间是固定的，很多客户端一起断线时不够稳妥

现在客户端断线后的重连时间是固定的：

- 前 5 分钟每 3 秒一次
- 之后每 10 秒一次

这个逻辑简单，但如果大量客户端同时掉线，就可能在同一时间一起重连，给服务端带来较大的瞬时压力。

### 7.4 服务端核心文件承担的事情过多，后续修改互相影响的风险较高

这个问题不是说当前代码一定错了，而是代码边界不够清楚。

现在最核心的几个文件里，同时放了很多不同性质的事情，例如：

- 启动和关闭
- 路由
- WebSocket 连接
- 管理接口
- 状态整理
- 隧道动作处理
- 写入存储
- 页面通知

这样做短期内能工作，但随着功能继续增加，会越来越容易出现：

- 改一处影响多处
- 代码审查困难
- 新同事不容易接手

### 7.5 关闭流程还依赖固定等待时间

当前服务端和客户端关闭时，仍然有固定等待时间，例如先等待一小段时间，再继续清理。

这种方式不是完全不能用，但它的问题很直接：

- 某些环境下可能等得不够
- 某些环境下又会等得太保守
- 本质上还是在靠经验值，而不是明确的退出完成信号

## 8. 后续治理项

### 8.1 自动检查还缺少并发相关检查

现在自动检查流程已经做了不少事情：

- 前端代码检查
- 前端构建
- Go vet
- Go test
- 端到端测试
- 交叉编译

但还没有把并发相关检查纳入进去。

这个项目里有不少并发场景，例如：

- goroutine
- WebSocket
- 多条连接同时关闭和重连

这类问题平时不一定稳定复现，所以越晚补检查，后面越难追。

### 8.2 README 里的 Go 版本说明和仓库实际要求还不一致

现在的情况是：

- `go.mod` 写的是仓库实际要求
- 自动检查流程也是跟着 `go.mod` 走
- `README` 里展示的 Go 版本说明和它并不一致

所以真正的问题不是“自动检查流程和代码不一致”，而是：

> **README 的说明和仓库实际要求不一致。**

### 8.3 TCP / UDP 隧道本身在长时间运行、重复失败、慢连接等场景下，处理还不够完整

当前这里后续要解决的，主要是“隧道本身还能不能继续工作”。

不打算放进这一项里的内容包括：

- 本地服务是否可用
- 第一笔真实流量是否成功
- 最近成功率是多少

更适合继续整理的是这些隧道本身的问题：

- TCP 监听长期运行是否稳定
- UDP 主读循环出错后如何处理
- 服务端到客户端的转发通道持续失败时怎么办
- 长连接长期占用、慢连接、空闲过久时如何处理
- UDP 会话保留、清理、总量上限、单 IP 上限是否合适

也就是说，这一项后续关注的是：

> **隧道这套链路本身的长期运行规则是否足够清楚。**

而不是去回答“本地服务是不是健康”或者“最近成功率是多少”。

### 8.4 还缺少更系统的端到端验证和手工验证记录

现在虽然已经有：

- 单元测试
- 一部分集成测试
- 一部分端到端测试
- 一部分批次级验证记录（例如 `docs/2026-03-27-batch-c-validation.md`、`docs/2026-03-27-console-snapshot-validation.md`）

但还不够覆盖下面这些更接近真实使用的问题：

- 三类隧道是否都能长期稳定工作
- 创建成功后，第一笔真实流量是否一定能通过
- 慢连接、长连接、失败重试等场景表现是否合理
- TCP / UDP 是否有足够的真实链路验证

换句话说，现在“能跑通”和“已经证明足够稳”之间，仍然有距离。

## 9. 关于状态数据的后续整理方向

### 9.1 继续保留的方向

下面这些方向，当前代码已经基本是这样做的，后续应继续保持：

1. **客户端继续主动上报机器状态**  
   不依赖页面是否在线。

2. **服务端继续作为统一汇总层**  
   页面只读取服务端已经掌握的数据。

3. **页面继续只承担展示职责**  
   不负责决定客户端什么时候去采集机器状态。

### 9.2 需要补上的内容

后续建议继续沿用已经补上的“主动获取当前完整状态”入口（当前为 `GET /api/console/snapshot`）。

它的用途不是替代所有现有接口，而是用于：

- 页面第一次进入
- 页面重新连上之后
- 用户手动刷新
- 将来系统有能力判断当前页面状态可能不可信的时候

另外，后续还应把下面这些时间补清楚：

- 服务端状态是什么时候整理出来的
- 客户端最近一次在线是什么时候确认的
- 客户端机器状态最近一次上报是什么时候
- 当前这份完整状态是什么时候生成的

### 9.3 现在还不能直接做的事

在下面这两个条件补齐之前，不建议直接去掉当前“定时再发一次完整状态”的做法：

1. 页面能知道自己是不是漏掉了变化通知
2. 服务端和页面之间有更清楚的“重新拿一份完整状态”的规则

也就是说，后续更合理的推进顺序应该是：

- 先把状态口径写清楚
- 继续把主动获取完整状态的入口稳定下来并扩展使用场景
- 再补判断页面是否漏掉通知的办法
- 最后再考虑是否移除当前的定时完整状态

3. TCP `Accept()` 异常退出后只打日志、不改状态的问题已收口

## 11. 总结

总的来说，当前项目已经具备核心功能，也已经修掉了一部分前期比较明显的问题。

现在还剩下的问题，主要集中在五件事上：

1. **重要数据写文件失败后，系统不一定能及时明确地报错**
2. **某些异常场景下，系统会继续运行，但状态已经不可靠**
3. **某些真实失败还没有完整反映到隧道状态和页面上**
4. **页面同步状态的方式还能继续改得更清楚、更稳妥**
5. **后续继续迭代时，代码边界和验证手段还需要加强**

这份文档的目标不是一次性解决全部问题，而是先把当前系统的状态边界和待处理问题说明白，后面再逐项推进。
